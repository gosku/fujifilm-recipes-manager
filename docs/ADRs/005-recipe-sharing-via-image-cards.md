# ADR 005 — Recipe sharing via image cards with embedded QR codes

**Status**: Accepted
**Date**: 2026-04-18

---

## Context

Fujifilm film simulation recipes are stored in EXIF metadata. When a user shares a photo on Reddit, Instagram, or Twitter/X, the platform strips all EXIF data during upload. The recipe is lost and cannot be recovered from the shared image.

Users in the Fujifilm community regularly share recipes alongside their photos, but doing so requires posting the recipe as a separate text block — a manual, error-prone step that breaks the connection between the image and the settings that produced it.

---

## Problem

How can a user share a recipe alongside a photo in a way that:

1. Survives EXIF stripping by social platforms.
2. Is machine-readable so the recipe can later be re-imported into the app.
3. Is human-readable without any tooling.
4. Does not depend on a proprietary format or a centralised service.

---

## Options considered

### Option A — Text overlay on the original photo

Burn the recipe field/value pairs as text directly onto the user's photo and share that.

**Why we did not choose this option:**

A text overlay satisfies human readability but not machine readability. Re-importing a recipe from such an image would require OCR-ing the entire overlay — fragile, layout-dependent, and error-prone. OCR must infer characters from pixel shapes, making it sensitive to font choice, point size, contrast, rotation, antialiasing, and image compression artefacts. Even high-quality engines produce errors on stylised or small labels. The goal is to satisfy both concerns: a human can read the card, and a machine can import the recipe reliably.

### Option B — Dedicated recipe card image

Generate a separate 1080×1080 image that combines both concerns: a human-readable text overlay listing the recipe fields, and a QR code that encodes the recipe for machine import. The user can share the card alongside their photo, or on its own when the focus is on the recipe.

**Why this was chosen:** it satisfies both readability and importability in a single artefact, leaves the original photo untouched, and the fixed canvas gives full control over layout and legibility. The QR code is also a significantly more reliable machine-readable format than OCR of text: it is a structured binary symbol with built-in error correction (up to 30% of the code can be damaged and still decode correctly), making it robust to image compression, cropping, and scaling that social platforms routinely apply to uploaded images.

### Option C — External link / QR to a hosted page

Generate a short URL encoding the recipe and overlay it on the image as a QR code. The QR points to a page that displays the recipe.

**Why we did not choose this option:** requires a live server and a persistent URL scheme. This project runs locally with no external hosting requirement. A QR code pointing to a dead URL after a server change is worse than no QR code at all.

---

## Decision

Generate a **recipe card**: a 1080×1080 JPEG composed of a background layer (user photo or gradient), a semi-transparent text panel with the recipe's field/value pairs, and a QR code in the bottom-right corner that encodes the recipe as compact JSON.

Users can upload recipe-card images back to the app to **import the recipe**: the QR code is decoded and the recipe is stored in the database, deduplicated against existing recipes with identical settings.

---

## QR payload format

The QR code encodes the recipe as **minified JSON with long snake_case keys** that match the model field names exactly. The keys are self-explanatory to any reader without consulting a legend:

```json
{
  "v": 1,
  "name": "Golden Hour",
  "film_simulation": "Classic Chrome",
  "grain_roughness": "Weak",
  "grain_size": "Small",
  "color_chrome_effect": "Off",
  "color_chrome_fx_blue": "Off",
  "white_balance": "Daylight",
  "white_balance_red": 0,
  "white_balance_blue": 0,
  "dynamic_range": "200",
  "d_range_priority": "Off",
  "highlight": -1,
  "shadow": 1,
  "color": 0,
  "sharpness": -1,
  "high_iso_nr": -2,
  "clarity": 0
}
```

An open format was chosen over a proprietary encoding. The goals of openness are:

- Any developer can decode cards generated by this app without reverse-engineering.
- Cards generated by compatible third-party tools can be imported here.
- The recipe is self-describing — a plain JSON viewer is sufficient to read it.

### Schema versioning

The `"v"` field is present in every payload from the first version. The reader hard-rejects any version other than `1` with an `unsupported_version` error, so a future breaking change can bump the version and both old and new cards remain unambiguous.

### Null / inapplicable field omission

A field is omitted from the JSON payload only when its value is `None` **and** the field is not applicable to the recipe's film simulation (e.g., colour fields for monochromatic film simulations, monochrome tone fields for colour simulations). Fields whose value is `0` or any other default are always included — they are meaningful settings, not absent ones. The `grain_size` field is additionally omitted when `grain_roughness` is `"Off"`.

### Name field

The `"name"` key is included only when the recipe's name is non-empty. Nameless recipes produce byte-identical payloads to a card generated before this key existed, preserving deduplication.

### Decimal field encoding

Decimal fields (`highlight`, `shadow`, `color`, `sharpness`, `high_iso_nr`, `clarity`, `monochromatic_color_warm_cool`, `monochromatic_color_magenta_green`) are stored as signed strings in the database (`"+2"`, `"-1.5"`, `"0"`). In the JSON payload they are encoded as JSON numbers — integer when the value is whole (`-1`, `0`, `2`), float otherwise (`1.5`, `-1.5`) — to keep the payload compact and type-correct for consumers that do not know the app's internal string convention.

## Card composition

Card generation is driven by a **template system**. Each `CardTemplate` defines the visual rules for composing the card — label style, background effect, and output dimensions. New templates can introduce entirely different visual approaches (layout, typography, colour scheme, panel placement) without touching the composition pipeline. Additional templates are planned.

The current default templates compose the card in layers using **Pillow**:

1. **Background** — either a cover-fill resize/crop of a user photo (optionally Gaussian-blurred), or a soft dark teal→indigo gradient when no photo is selected.
2. **Semi-transparent panel** — an RGBA overlay on the left half of the canvas, composited with `Image.alpha_composite`, to ensure text legibility regardless of background content.
3. **Text overlay** — recipe field/value lines rendered with `ImageFont`. Two label styles are offered: _full labels_ (`White Balance`, `Highlights`, …) and _short labels_ (`WB`, `HL`, …). Field order matches the recipe detail view for consistency.
4. **QR code** — generated with `qrcode[pil]`, resized to 200×200 px, pasted 20 px from the bottom-right corner.

### Background effect options

Two background effects are available when a photo is selected:

- **Blur** — Gaussian blur softens the photo, making text readable without the panel needing to be opaque.
- **Sharp** — the original photo at full sharpness, relying solely on the semi-transparent panel.

When no photo is selected the gradient is always used — blur is not applicable.

### EXIF embedding on gradient cards

When the background is a gradient (no real photo), the recipe JSON is additionally embedded in the JPEG EXIF `UserComment` field using **piexif**. This is a forward-compatibility measure: if a future version supports EXIF-based import, gradient cards produced today will already carry the payload. The current importer reads only the QR code; EXIF fallback is not implemented.

## Live preview

The modal shows a live preview of the card as options are changed, generated from the same composition pipeline but written to a deterministic path in `/tmp/` rather than saved to the database. The preview is regenerated on every form change via HTMX without a page reload. A permanent `RecipeCard` record is only created (and the file saved to `RECIPE_CARDS_DIR`) when the user explicitly clicks **Create Recipe Card**.

---

## Persistence

A `RecipeCard` model stores:

- `filepath` — path to the saved JPEG.
- `template` — label style + background effect encoded as a string key (e.g. `long_label`, `short_label_sharp`).
- `image` — optional FK to the background `Image`.
- `recipe` — FK to the `FujifilmRecipe`.

A recipe can have multiple cards (different templates, different background images). The `RecipeCard` record is the authoritative link between a recipe and its generated card files.

## Reading and importing from QR cards

### QR decoding

Cards are imported by uploading one or more recipe-card images through the **Import from recipe cards** option in the Recipes Explorer dropdown. Each uploaded file is written to a temporary file under `/tmp/`, the QR is decoded, and the temporary file is deleted immediately — whether the operation succeeds or fails.

The QR is decoded using `opencv-python-headless` (`cv2.QRCodeDetector`). A 200 px QR embedded in a 1080 px card is too small for a single-pass `detectAndDecode()` call to decode reliably. A **two-pass strategy** is used:

1. Attempt `detectAndDecode()` directly on the full image.
2. If that fails, run `detect()` to locate the QR region, crop it with a small padding, upscale the crop 3× with bicubic interpolation, and attempt `detectAndDecode()` on the upscaled crop.

This approach is reliable across both card templates and survives the image compression that social platforms apply.

### Schema validation

After decoding, the JSON payload is validated in order:

1. The decoded string must be valid JSON and the top-level value must be a dict.
2. `"v"` must be present, must be an integer, and must equal `1` — any other value raises `InvalidQRRecipePayloadError(reason="unsupported_version")`.
3. All keys in the payload must belong to the known schema — any unknown key raises `InvalidQRRecipePayloadError(reason="unknown_fields")`.
4. Each present field must have the expected type (`str`, `int`, or `int | float`) — a mismatch raises `InvalidQRRecipePayloadError(reason="type_mismatch")`. `bool` is explicitly rejected for numeric fields because Python's `isinstance(True, int)` returns `True`.
5. Required fields (`film_simulation`, `grain_roughness`, `d_range_priority`, `white_balance`, `white_balance_red`, `white_balance_blue`) must be present — a missing required field raises `InvalidQRRecipePayloadError(reason="type_mismatch")` (surfaced as an attrs construction error).

If no QR code can be decoded from the file, `QRCodeNotFoundError` is raised.

### Translation

`get_recipe_data_from_qr_recipe` translates a decoded `QRFujifilmRecipe` back to a `FujifilmRecipeData`, inverting the encoding decisions made at card-generation time:

- Decimal fields go from JSON number (`-1`, `1.5`) back to signed string (`"-1"`, `"+1.5"`).
- `grain_size` defaults to `"Off"` when absent and `grain_roughness` is `"Off"`, matching the create-side default.
- `color_chrome_effect` and `color_chrome_fx_blue` default to `""` when absent (they are omitted for monochromatic simulations), matching how those fields are stored for monochrome recipes.

### Deduplication

Two cards with identical recipe settings produce the same `FujifilmRecipe` record. Uniqueness is defined by the full set of recipe settings fields; `name` is excluded from the uniqueness constraint. Uploading the same card twice returns the same record both times.

### Name persistence

The recipe name from the QR payload is passed to `get_or_create` via Django's `defaults` mechanism. This means:

- On **first import** — the name from the card is stored.
- On **re-import** of an identical card — the existing name is preserved; the name from the card is ignored. This matches the behaviour of all other recipe imports (EXIF import also never overwrites an existing recipe's name).

### Per-file failure handling

Upload failures are handled per-file. When a file cannot be decoded or its payload is invalid, the filename is added to the `failed` list of the result and processing continues with the remaining files. A partial success (some files imported, some failed) is a valid outcome.

Each failure also publishes a structured `recipe.import.qr_card.failed` event carrying the `filename` and a `failure_reason` string (`"qr_not_found"`, `"unsupported_version"`, `"unknown_fields"`, `"type_mismatch"`, `"invalid_json"`). This makes failures visible in the dev terminal and the events log without relying on the HTTP response.

### Import UI

The Recipes Explorer dropdown has two import openers alongside the existing EXIF-image opener:

- **Import from photos** — uploads images and reads EXIF metadata.
- **Import from recipe cards** — uploads card images and reads the QR code.

Both openers share a single modal and form. Each opener button carries `data-import-url`, `data-import-title`, and `data-import-desc` attributes. A shared JavaScript handler copies these onto the form's `hx-post` attribute, the modal heading, and the modal description when the modal opens. `htmx.process(form)` is called after setting `hx-post` so that HTMX re-reads the attribute — HTMX 2.x caches its config at page load and ignores subsequent `setAttribute` calls otherwise.

## Consequences

- Recipe cards are generated on-demand and stored locally in `RECIPE_CARDS_DIR` (configurable via the `RECIPE_CARDS_DIR` environment variable, defaulting to `BASE_DIR/recipe_cards`).
- The QR payload schema is pinned to `"v": 1`. Any breaking change to the key set or value types must bump the version so old and new cards can be read simultaneously.
- Name is stored on first import and never overwritten on re-import. A recipe's canonical name is the one set when it was first created (whether via EXIF import, QR import, or manual entry).
- EXIF `UserComment` is embedded in gradient cards as a forward-compatibility measure but is not read by the current importer. If EXIF-based import is added in a future version, all existing gradient cards will already carry the payload.
- `UploadedFile` and `ImportRecipesResult` live in the domain layer (`src/domain/recipes/dataclasses.py`) so both the EXIF usecase and the QR-card usecase can share them without a cross-usecase import.
