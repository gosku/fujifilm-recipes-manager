# ADR 005 — Recipe sharing via image cards with embedded QR codes

**Status**: Accepted
**Date**: 2026-04-18

---

## Context

Fujifilm film simulation recipes are stored in EXIF metadata. When a user shares a photo on Reddit, Instagram, or Twitter/X, the platform strips all EXIF data during upload. The recipe is lost and cannot be recovered from the shared image.

Users in the Fujifilm community regularly share recipes alongside their photos, but doing so requires posting the recipe as a separate text block — a manual, error-prone step that breaks the connection between the image and the settings that produced it.

---

## Problem

How can a user share a recipe alongside a photo in a way that:

1. Survives EXIF stripping by social platforms.
2. Is machine-readable so the recipe can later be re-imported into the app.
3. Is human-readable without any tooling.
4. Does not depend on a proprietary format or a centralised service.

---

## Options considered

### Option A — Text overlay on the original photo

Burn the recipe field/value pairs as text directly onto the user's photo and share that.

**Why we did not choose this option:**

A text overlay satisfies human readability but not machine readability. Re-importing a recipe from such an image would require OCR-ing the entire overlay — fragile, layout-dependent, and unnecessary when a QR code can carry the same data reliably. The goal is to satisfy both: a human can read the card, and a machine can import the recipe by scanning the QR.

### Option B — Dedicated recipe card image

Generate a separate 1080×1080 image that combines both concerns: a human-readable text overlay listing the recipe fields, and a QR code that encodes the recipe for machine import. The user can share the card alongside their photo, or on its own when the focus is on the recipe.

This is the approach used by the Fujifilm community on social media — standalone "recipe cards" are already a recognised genre.

**Why this was chosen:** it satisfies both readability and importability in a single artefact, leaves the original photo untouched, and the fixed canvas gives full control over layout and legibility.

### Option C — External link / QR to a hosted page

Generate a short URL encoding the recipe and overlay it on the image as a QR code. The QR points to a page that displays the recipe.

**Why we did not choose this option:** requires a live server and a persistent URL scheme. This project runs locally with no external hosting requirement. A QR code pointing to a dead URL after a server change is worse than no QR code at all.

---

## Decision

Generate a **recipe card**: a 1080×1080 JPEG composed of a background layer (user photo or gradient), a semi-transparent text panel with the recipe's field/value pairs, and a QR code in the bottom-right corner that encodes the recipe as compact JSON.

---

## QR payload format

The QR code encodes the recipe as **minified JSON with short, self-explanatory keys**:

```json
{
  "film_sim": "Classic Chrome",
  "wb": "Daylight",
  "wb_r": 2,
  "wb_b": -1,
  "hl": -1,
  "sh": 1,
  "color": 0,
  "sharp": -1,
  "nr": -2,
  "dr": 200
}
```

An open format was chosen over a proprietary encoding. The goals of openness are:

- Any developer can decode cards generated by this app without reverse-engineering.
- Cards generated by compatible third-party tools can be imported here.
- The recipe is self-describing — a plain JSON viewer is sufficient to read it.

A `version` field can be added in future schema iterations to allow the format to evolve without breaking existing codes.

### Null / inapplicable field omission

A field is omitted from the JSON payload only when its value is `None` **and** the field is not applicable to the recipe's film simulation (e.g., colour fields for monochromatic film simulations, monochrome tone fields for colour simulations). Fields whose value is `0` or any other default are always included — they are meaningful settings, not absent ones.

---

## Card composition

Card generation is driven by a **template system**. Each `CardTemplate` defines the visual rules for composing the card — label style, background effect, and output dimensions. New templates can introduce entirely different visual approaches (layout, typography, colour scheme, panel placement) without touching the composition pipeline. Additional templates are planned.

The current default templates compose the card in layers using **Pillow**:

1. **Background** — either a cover-fill resize/crop of a user photo (optionally Gaussian-blurred), or a soft dark teal→indigo gradient when no photo is selected.
2. **Semi-transparent panel** — an RGBA overlay on the left half of the canvas, composited with `Image.alpha_composite`, to ensure text legibility regardless of background content.
3. **Text overlay** — recipe field/value lines rendered with `ImageFont`. Two label styles are offered: _full labels_ (`White Balance`, `Highlights`, …) and _short labels_ (`WB`, `HL`, …). Field order matches the recipe detail view for consistency.
4. **QR code** — generated with `qrcode[pil]`, resized to 200×200 px, pasted 20 px from the bottom-right corner.

### Background effect options

Two background effects are available when a photo is selected:

- **Blur** — Gaussian blur softens the photo, making text readable without the panel needing to be opaque.
- **Sharp** — the original photo at full sharpness, relying solely on the semi-transparent panel.

When no photo is selected the gradient is always used — blur is not applicable.

### EXIF embedding on gradient cards

When the background is a gradient (no real photo), the recipe JSON is additionally embedded in the JPEG EXIF `UserComment` field using **piexif**. This enables future import: a user can feed the gradient card back to the app and recover the recipe without scanning the QR. This embedding is skipped for photo-backed cards to avoid polluting the photo's EXIF with recipe data.

---

## Live preview

The modal shows a live preview of the card as options are changed, generated from the same composition pipeline but written to a deterministic path in `/tmp/` rather than saved to the database. The preview is regenerated on every form change via HTMX without a page reload. A permanent `RecipeCard` record is only created (and the file saved to `RECIPE_CARDS_DIR`) when the user explicitly clicks **Create Recipe Card**.

---

## Persistence

A `RecipeCard` model stores:

- `filepath` — path to the saved JPEG.
- `template` — label style + background effect encoded as a string key (e.g. `long_label`, `short_label_sharp`).
- `image` — optional FK to the background `Image`.
- `recipe` — FK to the `FujifilmRecipe`.

A recipe can have multiple cards (different templates, different background images). The `RecipeCard` record is the authoritative link between a recipe and its generated card files.

---

## Consequences

- Recipe cards are generated on-demand and stored locally in `RECIPE_CARDS_DIR` (configurable via the `RECIPE_CARDS_DIR` environment variable, defaulting to `BASE_DIR/recipe_cards`).
- The QR payload format is versioned implicitly by its key names; a `"v"` field should be added before any breaking schema change.
- Part 2 — scanning a recipe card or QR code to import a recipe — is deferred and not part of this implementation.
